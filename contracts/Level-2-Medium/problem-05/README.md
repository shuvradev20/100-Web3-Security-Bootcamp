# Problem 5: Off-Chain Signature Airdrop Claim 🪂

## 📝 Real-world Scenario
A DeFi protocol needs to distribute promotional airdrop tokens to whitelisted users efficiently. To save massive gas costs associated with on-chain whitelist storage, the protocol uses off-chain cryptographic signatures from an admin, allowing users to claim their tokens securely and seamlessly.

## ⚙️ Business Logic
- The contract is initialized with the address of the ERC20 token to be distributed and assigns the deployer as the admin.
- The admin generates an off-chain signature approving a specific user for a specific token amount.
- Users call the `claim` function, providing the approved amount and the admin's signature.
- The contract cryptographically verifies that the signature was generated by the admin and specifically intended for the caller.
- Upon successful verification, the designated amount of ERC20 tokens is safely transferred to the user.

## 🛡️ Security & 2026 Standards Implemented
- **ECDSA Signature Verification:** Utilized OpenZeppelin's `ECDSA` and `MessageHashUtils` for secure, industry-standard cryptographic signature recovery (`recover`).
- **Anti-Signature Replay (Double Dipping):** Implemented a `checkClaim` mapping to track users who have already claimed, permanently preventing a single valid signature from being reused.
- **Cross-Chain & Cross-Contract Protection:** Included `block.chainid` and `address(this)` inside the hashed message payload. This completely prevents attackers from replaying the same signature on a fork, a different network, or a different instance of the contract.
- **Front-Running Prevention:** Bound `msg.sender` directly into the hashed message. If a malicious MEV bot intercepts the transaction in the mempool and tries to use the signature, the transaction will revert because the signature is uniquely cryptographically tied to the intended user's address.
- **Strict CEI Pattern:** Updated the state (`checkClaim[msg.sender] = true;`) *before* executing the external ERC20 token transfer, eliminating potential reentrancy attack vectors.
- **Custom Errors:** Used gas-efficient custom errors (`AlreadyClaimed()`, `InvalidSignature()`, `TransferFailed()`, `ZeroAddress()`) instead of expensive `require` strings.
